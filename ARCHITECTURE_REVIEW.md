# Architecture Review Report

## 1. Введение
Данный отчет представляет собой анализ архитектуры приложения (Next.js, Drizzle ORM, PostgreSQL) на предмет соответствия принципам Clean Architecture (в прагматичном ключе) и масштабируемости (как кодовой базы, так и технической производительности).

Приложение спроектировано с использованием слоистой архитектуры (Domain, Repositories, Services, UI/API) с базовым Dependency Injection через `src/lib/container.ts`.

## 2. Анализ на соответствие Clean Architecture

### Позитивные моменты:
- **Разделение слоев:** Наличие директорий `domain`, `repositories`, `services`, `app` четко показывает намерение отделить бизнес-логику от деталей реализации.
- **Dependency Injection:** Использование `src/lib/container.ts` для сборки зависимостей (сервисы получают репозитории через конструктор). Это отличный фундамент для тестируемости (можно мокать репозитории).

### Нарушения и зоны для улучшения:
- **Протечки БД в API слой:** В некоторых API роутах (например, `src/app/api/admin/sessions/route.ts`, `src/app/api/admin/universities/bulk/route.ts`, `src/app/api/seed/route.ts` и `src/app/api/test/explain/route.ts`) происходит прямое обращение к `db` и схемам `drizzle`. Это нарушает правило изоляции слоев, так как API-слой должен общаться только с Service-слоем.
- **God Components (Монолитные компоненты UI):** Страницы, такие как `src/app/admin/questions/page.tsx` и `src/app/tests/page.tsx`, содержат огромное количество логики (стейт, рендеринг, запросы, модальные окна, форматирование). Это нарушает Single Responsibility Principle (SRP) на уровне компонентов.
- **Смешение типов:** DTO (Data Transfer Objects) и доменные модели часто перемешаны. В идеале UI/API слой не должен знать о структурах базы данных напрямую.

## 3. Масштабируемость (Scalability)

### 3.1. Масштабируемость кодовой базы (Maintainability)
- **Структура фичей (Feature-Sliced Design / Vertical Slicing):** Текущая структура разделена по техническим слоям (`repositories`, `services`, `domain`). По мере роста проекта это приведет к тому, что для добавления одной фичи (например, "Система достижений") придется менять файлы в 4-5 разных папках. Более масштабируемым подходом является группировка по фичам/доменам (например, `src/modules/users/`, `src/modules/tests/`), где внутри каждой фичи лежат свои сервисы, репозитории и компоненты.
- **Управление состоянием UI:** Использование локального состояния (useState/useEffect) для данных, получаемых с сервера, делает код хрупким. Отсутствует библиотека для управления серверным состоянием (React Query или SWR).

### 3.2. Техническая масштабируемость и производительность
- **Отсутствие кэширования:** В проекте (на уровне API и компонентов) практически не используются механизмы кэширования Next.js (`unstable_cache`, `fetch` с revalidate) или внешние системы (Redis). Для платформы тестов, где вопросы читаются очень часто, отсутствие кэширования создаст высокую нагрузку на PostgreSQL при росте пользователей.
- **N+1 запросы и оптимизация БД:**
  - В `TestService.submitTest` происходит вызов репозитория `getQuestionById` в цикле `for...of`:
    ```typescript
    for (const answer of answers) {
      const question = await this.testsRepository.getQuestionById(answer.questionId);
      // ...
    }
    ```
    Это классическая проблема N+1. Нужно загружать все необходимые вопросы одним запросом `inArray`.
- **Массовые операции (Bulk operations):** В `src/app/api/admin/universities/bulk/route.ts` используется цикл для вставки университетов по одному:
  ```typescript
  for (let i = 0; i < body.length; i++) {
    // await db.insert(...)
  }
  ```
  Это медленно. Drizzle поддерживает массовую вставку массивом за один запрос к БД.
- **Отсутствие пагинации при тяжелых запросах:** Нужно убедиться, что все списочные запросы в админке или истории имеют жесткие лимиты.

## 4. Рекомендации по исправлению (Action Plan)

### Рефакторинг архитектуры (Clean Architecture)
1. **Изолировать БД:** Перенести все прямые вызовы `db.select()`, `db.insert()` из папки `src/app/api/` в соответствующие сервисы и репозитории. API роуты должны только парсить Request, вызывать методы Service и возвращать Response.
2. **Разбить God Components:** Вынести модальные окна, логику генерации и сложный UI из `admin/questions/page.tsx` и `tests/page.tsx` в отдельные мелкие компоненты (`src/components/features/...`).

### Улучшение производительности (Performance & Tech Scalability)
3. **Устранить проблему N+1 в тестах:** Переписать метод `submitTest` в `TestService`, чтобы он получал все вопросы списком до цикла проверки:
   ```typescript
   const questionIds = answers.map(a => a.questionId);
   const questions = await this.testsRepository.getQuestionsByIds(questionIds);
   // затем создать мапу для быстрого доступа и проверить ответы
   ```
4. **Оптимизировать Bulk Import:** В админке переписать импорты на использование `await db.insert(table).values(arrayOfData)`.
5. **Внедрить кэширование:** Использовать Next.js кэширование для публичных данных (например, список университетов, статические предметы) и/или добавить React Query (`@tanstack/react-query`) на клиенте для кэширования, дедупликации запросов и оптимистичных обновлений.

### Улучшение кодовой базы
6. **Организация директорий:** Рассмотреть постепенный переход к Feature-based архитектуре для новых модулей.
7. **Обработка ошибок:** Создать единый Middleware или утилиту для обработки ошибок в API, чтобы избежать дублирования блоков `try/catch`.
