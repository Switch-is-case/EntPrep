# Отчет о проверке проекта на соответствие принципам Clean Code

Проект представляет собой приложение на Next.js (App Router) с использованием React, TypeScript, Drizzle ORM и PostgreSQL. В целом, проект имеет неплохую начальную структуру, включая разделение на слои (`domain`, `repositories`, `services`, `app`), но содержит ряд проблем, нарушающих принципы чистого кода (Clean Code) и лучшие практики разработки.

Ниже приведен подробный разбор найденных проблем и рекомендации по их устранению.

## 1. Архитектура и структура (Architecture & Structure)

### 1.1 Отсутствие Dependency Injection (DI)
В слоях `services` и `repositories` создаются и сразу экспортируются экземпляры классов:
```typescript
// src/repositories/users.repository.ts
export const usersRepository = new UsersRepository();
```
* **Проблема:** Это создает жесткую связанность (tight coupling) между сервисами и репозиториями. Если потребуется написать unit-тесты для сервисов, замокать базу данных (репозиторий) будет сложно.
* **Решение:** Использовать паттерн Dependency Injection. Передавать репозитории в сервисы через конструктор.

### 1.2 Логика в компонентах и дублирование
* **Проблема:** В клиентских компонентах (особенно в `src/app/admin/*`, `src/app/history/*`) размазана логика работы с API (вызовы `fetch`, обработка токенов авторизации).
* **Решение:** Вынести работу с API в отдельные кастомные хуки (например, `useFetchUsers`) или использовать библиотеки типа `SWR` / `React Query`. Это позволит уменьшить дублирование кода, упростить компоненты и улучшить управление состоянием загрузки и ошибок.

### 1.3 База данных и ORM (Drizzle)
В целом схема данных в `src/db/schema.ts` выглядит нормально, но:
* В репозиториях повсеместно используется приведение типов `as unknown as Type`, например: `return user as unknown as User;` (в `UsersRepository`).
* **Проблема:** Это обход (escape hatch) системы типов TypeScript, скрывающий реальные ошибки несовпадения типов БД и доменных моделей.
* **Решение:** Использовать возможности Drizzle по выведению типов моделей (`typeof table.$inferSelect`).

## 2. Типизация и TypeScript

В проекте есть ошибки компиляции (`npx tsc --noEmit`), что является критическим нарушением:

### 2.1 Неявный тип `any[]`
* В `src/repositories/universities.repository.ts` переменные `insertedPrograms` и `currentPrograms` инициализируются как `[]`, что приводит к ошибке `implicitly has an 'any[]' type`.
* **Решение:** Указать точный тип: `let insertedPrograms: typeof universityPrograms.$inferSelect[] = [];`.

### 2.2 Чрезмерное использование `any`
В коде встречается много `any`, особенно в обработчиках ошибок `catch (error: any)`.
* **Проблема:** Использование `any` отключает TypeScript и лишает код безопасности типов.
* **Решение:** Использовать `catch (error: unknown)` с последующей проверкой (type narrowing), например: `if (error instanceof Error) { ... }`.

## 3. Проблемы React и ESLint

Запуск линтера выявляет ряд серьезных ошибок и предупреждений, связанных с React Hooks.

### 3.1 Вызов `setState` синхронно в `useEffect`
* В файлах `admin/universities/page.tsx`, `admin/users/page.tsx`, `profile/page.tsx`, `components/Providers.tsx` встречается ошибка `react-hooks/set-state-in-effect`.
* **Проблема:** Синхронный вызов сеттера внутри `useEffect` приводит к каскадным перерендерам (cascading renders) и проблемам с производительностью.
* **Решение:** Либо перенести логику инициализации (как в `Providers`), либо убедиться, что состояние обновляется только при реальных изменениях. Часто это признак того, что `useEffect` используется там, где он не нужен (например, получение данных при загрузке лучше выносить в хуки React Query или управлять ими через асинхронные функции без синхронных сетов при монтировании, если это не обосновано).

### 3.2 Отсутствующие зависимости в `useEffect` / `useCallback`
* **Проблема:** Ошибки `react-hooks/exhaustive-deps`. Массивы зависимостей не содержат всех переменных, используемых внутри хука. Это классическая проблема, которая приводит к stale closures (замыканиям на устаревшие данные) или бесконечным циклам.
* **Решение:** Внимательно заполнить массивы зависимостей (линтер подсказывает, какие переменные пропущены). Для избежания циклов использовать `useCallback` для оборачивания функций.

### 3.3 Мутация `window.location.href` во время рендера
В `src/app/register/page.tsx`:
```typescript
if (user) {
  window.location.href = "/profile";
  return null;
}
```
* **Проблема:** Прямая модификация сайд-эффекта во время рендеринга (ошибка `react-hooks/immutability`).
* **Решение:** Использовать компонент перенаправления Next.js (`redirect` / `useRouter().push()`), и вызывать его либо в `useEffect`, либо в обработчике.

### 3.4 Неоптимизированные изображения
* Линтер предупреждает об использовании тега `<img>` вместо компонента `<Image>` из `next/image` в `universities/page.tsx` и `history/[id]/page.tsx`.
* **Решение:** Заменить `<img src={...} />` на `<Image src={...} width={...} height={...} alt="..." />` для автоматической оптимизации.

## 4. Чистота Кода (Clean Code Principles)

* **Магические числа и строки:** В `QuestionsService` жестко зашито ограничение `dataArray.length > 500`. Такие значения лучше выносить в константы: `const MAX_IMPORT_SIZE = 500`.
* **Обработка ошибок:** Сейчас ошибки просто пробрасываются дальше как объекты `Error`, и API роуты проверяют сообщение по строке: `error.message === "Email already registered" ? 409 : 500`.
  * **Решение:** Использовать пользовательские классы ошибок (например, `ConflictError`, `ValidationError`) для более надежной проверки типов ошибок вместо сравнения строк.

## Итог

Проект нуждается в рефакторинге. В первую очередь необходимо:
1. Починить ошибки компиляции TypeScript (избавиться от `any` и неявных типов).
2. Исправить ошибки линтера React (зависимости хуков и `setState` в эффектах).
3. Ввести строгую типизацию для ответов от базы данных (избегать `as unknown as Type`).
4. Постепенно внедрить инъекцию зависимостей (DI) для облегчения тестирования.
