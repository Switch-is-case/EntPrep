# Отчет по архитектуре и качеству кода проекта (Clean Architecture & Clean Code)

Данный отчет содержит анализ текущей архитектуры проекта на соответствие принципам Clean Architecture и Clean Code, с учетом использования фреймворка Next.js (App Router) и прагматичного подхода.

## 1. Общая структура проекта (Clean Architecture)

Проект хорошо структурирован и разделен на логические слои. Текущая архитектура очень близка к классической Clean Architecture и правильно использует Dependency Injection.

**Наблюдаемая слоистая структура:**
- `src/domain` — Слой домена. Содержит интерфейсы (TypeScript Types) и DTO.
- `src/repositories` — Слой доступа к данным (Data Access). Инкапсулирует работу с Drizzle ORM.
- `src/services` — Слой бизнес-логики (Use Cases). Содержит основную логику приложения.
- `src/app/api` (и UI) — Слой представления и доставки (Delivery Layer). Роуты Next.js.
- `src/lib/container.ts` — Dependency Injection (DI) контейнер.

### Плюсы текущей архитектуры
✅ **Изоляция базы данных:** Код Drizzle ORM (запросы `db.select()`, таблицы из `schema.ts`) полностью изолирован в `src/repositories`. Сервисы и API-роуты ничего не знают о Drizzle.
✅ **Направление зависимостей (Dependency Rule):** Зависимости направлены внутрь. API зависит от Services, Services зависят от Repositories и Domain. Нет циклических зависимостей.
✅ **Использование DTO:** Для передачи данных между API и сервисами используются типизированные DTO из слоя Domain (`LoginDTO`, `CreateQuestionDTO` и т.д.).
✅ **Dependency Injection (DI):** Сервисы получают репозитории через конструктор (`constructor(private readonly questionsRepository: QuestionsRepository)`), а инстансы собираются в `container.ts`. Это делает код легко тестируемым (легко замокать репозитории).
✅ **Обработка ошибок:** Хорошая реализация кастомных ошибок (`AppError`, `ValidationError` и т.д.) в `src/lib/errors.ts`. Сервисы выбрасывают эти ошибки, а API-слой их перехватывает и трансформирует в правильные HTTP-статусы.

### Узкие места и потенциальные нарушения
⚠️ **Доменный слой слишком "анемичный" (Anemic Domain Model):** В `src/domain` лежат только типы TypeScript. В классической архитектуре там должны быть доменные сущности с поведением (классы с методами валидации или бизнес-правилами). В текущей (прагматичной) реализации вся бизнес-логика "размазана" по сервисам (например, валидация при импорте вопросов в `QuestionsService`).
*Однако, для CRUD-ориентированных приложений на Node.js / TypeScript анемичная модель — это частый и вполне допустимый компромисс.*

## 2. Анализ микро-уровня (Clean Code)

В целом код написан аккуратно, используется строгая типизация.

### Плюсы
✅ **Нейминг:** Имена переменных, методов и классов понятные и отражают их суть (`getQuestions`, `findAll`, `AuthService`).
✅ **DRY (Don't Repeat Yourself):** Логика извлечения `userId` вынесена в `getUserIdFromRequest` (например, в `src/app/api/test/start/route.ts`).
✅ **Отсутствие бизнес-логики в контроллерах:** API роуты (контроллеры) только принимают JSON, вызывают сервис и возвращают ответ.

### Минусы и нарушения
⚠️ **Принцип единой ответственности (SRP) в `TestService`:**
Метод `startTest` в `src/services/test.service.ts` выглядит немного перегруженным. Он занимается:
1. Проверкой пользователя.
2. Определением количества вопросов по предметам (захардкоженная логика `ENT_QUESTION_COUNTS`).
3. Сборкой вопросов из разных категорий.
В идеале, логику сборки конфигурации теста (какие предметы и сколько вопросов) можно вынести в отдельную доменную сущность или вспомогательный сервис/функцию.

⚠️ **Использование `any`:** В типах `Question` (`src/domain/questions/types.ts`) для массивов опций используется `string[] | any`. Использование `any` нарушает типобезопасность.
```typescript
optionsRu: string[] | any; // Стоит избегать any
```

⚠️ **Частичное дублирование SQL/Drizzle условий:** В репозиториях фильтрация собирается вручную (`conditions.push()`). В будущем, если фильтров станет много, это может стать нечитаемым.

## 3. Прагматичный подход и Next.js

Поскольку вы выбрали прагматичный подход, текущая архитектура — это **отличный баланс** между строгостью и удобством.

Вы не пошли по пути полного отказа от API-роутов в пользу Server Actions (что часто приводит к смешиванию базы данных и UI в Next.js App Router), а сохранили классическое разделение через DI-контейнер и сервисы. Это защитит проект при масштабировании.

## Рекомендации по улучшению (Рефакторинг)

1. **Избавиться от `any` в Domain:**
   В файле `src/domain/questions/types.ts` замените `string[] | any` на `string[]` (или `Json` тип из базы данных, если это JSONB).

2. **Вынос бизнес-правил из Сервисов:**
   Например, константы и правила формирования ЕНТ теста (количество вопросов по истории, математике) стоит держать в `src/domain/tests/rules.ts` или аналогичном файле, а не зашивать в `TestService`.

3. **Валидация DTO:**
   Сейчас в сервисах есть ручные проверки (например, `if (!data.email) throw new ValidationError(...)`). Рекомендуется интегрировать библиотеку вроде `zod` для автоматической валидации входящих данных на уровне API или в начале сервисов.

## Вывод
Проект **соответствует** принципам Clean Architecture в прагматичном понимании. Разделение ответственности (Separation of Concerns) выполнено на высоком уровне. Вы можете продолжать разработку в том же духе, проект не вызовет проблем с поддержкой и масштабированием в будущем.
